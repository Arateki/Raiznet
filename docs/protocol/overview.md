# Protocol Overview

Raiznet defines a layered protocol for decentralized crop monitoring. This page describes the full communication stack from the ESP32 sensor to the peer-to-peer mesh.

## Layers

```
┌──────────────────────────────────────────┐
│         Application (HTTP + P2P)         │
├──────────────────────────────────────────┤
│     Serialization  (Protobuf / nanopb)   │
├──────────────────────────────────────────┤
│  Transport  (HTTP  ·  ESP-NOW  ·  TCP)   │
├──────────────────────────────────────────┤
│           Identity  (Ed25519)            │
└──────────────────────────────────────────┘
```

## Wire format

All messages are serialized with **Protocol Buffers**:

- **Node.js / server**: `@bufbuild/protobuf` + code generated by `@bufbuild/protoc-gen-es`.
- **ESP32**: `nanopb` — a lightweight C implementation of the Protobuf encoding.

Using the same `.proto` schemas on both ends guarantees byte-level compatibility with no shared runtime.

## Transport by hop

| Hop | Protocol | Notes |
|---|---|---|
| ESP32 → ESP32 (mesh) | ESP-NOW | Same Wi-Fi channel, minimal CBOR payload |
| ESP32 → Server | HTTP POST | `POST /v1/telemetry`, JSON body with hex-encoded bytes |
| Server → Server | Hypercore / Hyperswarm TCP | Phase 2; Phase 1 is HTTP-only |

HTTP was chosen for the ESP32 → Server hop because sensors sleep for up to 30 minutes between readings. A persistent connection (WebSocket, MQTT) would be held open for nothing and would drain battery. Stateless HTTP POST is the correct model for infrequent, fire-and-forget ingest.

## Packet lifecycle

```
ESP32 reads sensors
  → assembles TelemetryBlock applying privacy policy
  → signs block with device Ed25519 secret key
  → POST /v1/telemetry to local server OR sends via ESP-NOW to gateway

Server receives TelemetryBlock
  → validates Ed25519 signature
  → checks seq monotonicity
  → routes public fields → raiznet_public.db
  → routes private fields → raiznet_private.db
  → (Phase 2) appends public block to device's Hypercore
  → Hyperswarm replicates Hypercore to peers
```

## Protobuf schemas

All `.proto` files live in `packages/protocol/proto/`. The generated TypeScript code is in `packages/protocol/src/gen/`.

| File | Contents |
|---|---|
| `telemetry.proto` | `SensorField`, `TelemetryBlock`, `TelemetryBatch` |
| `device.proto` | `Device`, `FieldPolicy`, `Disposition` enum |
| `user.proto` | `User`, `Contact` |

See [Proto Schemas](/reference/proto-schemas) for the full field reference.

## Identifiers

Every entity in the protocol is identified by its **Ed25519 public key** (32 bytes), typically serialized as hex in JSON and as raw bytes in Protobuf. There are no auto-incremented integers in the wire format.

| Entity | ID source |
|---|---|
| User | Public key derived from BIP-39 seed |
| Server | Public key generated at first boot |
| Device | Public key generated at provisioning |
| Filter / Catalog | Public key of its Hypercore |

## Sequence numbers

Each device maintains a monotonically increasing `seq` counter. On reconnection, the device asks the server for the last `seq` it received and retransmits only what is in the circular buffer above that point. Readings that aged out of the buffer before a sync can be recovered via direct BLE or serial connection to the device.
